Portal de gestión · v1.0.0

Baqueano — Reporte del proyecto

Sistema de gestión con arquitectura cliente-servidor desacoplada. Backend Spring Boot 3.5 sobre Java 25 LTS, frontend React 18 + Vite, persistencia en MySQL 8 con Flyway, empaquetado como WAR para WildFly 39 (Jakarta EE 10).

Java 25 LTS Spring Boot 3.5.14 React 18.3 Vite 5.4 MySQL 8 WildFly 39 92 tests verdes

1. Resumen ejecutivo

Baqueano es un portal de gestión interno con cuatro entidades principales administradas por ABMs completos (Usuarios, Perfiles, Parámetros y Contactos), autenticación JWT, autorización fina por perfil y un menú lateral dinámico construido en runtime desde la base de datos.

Backend
81 tests
Repos + servicios + controllers + integración
Frontend
11 tests
Vitest + React Testing Library en los forms
WAR
56 MB
SPA embebida, build en 38s

Capacidades principales

  • Login y logout con JWT HS256 + refresh tokens persistidos con hash SHA-256.
  • Bloqueo automático de usuario tras N intentos fallidos (parametrizable en BD).
  • Menú lateral 100% dinámico: cambia al modificar submenu_perfil sin redeploy.
  • Permisos finos por acción (VER / CREAR / EDITAR / ELIMINAR) chequeados en backend (@PreAuthorize) y reflejados en UI.
  • Alta pública de contactos (sin autenticación) desde formulario externo.
  • Observabilidad: actuator /health + /info, logs JSON estructurados en prod, OpenAPI/Swagger en dev.
  • SPA y API mismo origen en producción (un único WAR para WildFly).

3. Arquitectura del sistema

El sistema se despliega como un único WAR en WildFly 39 que sirve la SPA (React) y la API REST (Spring Boot) desde el mismo origen. En desarrollo se corre el backend embebido (Tomcat de Spring Boot) en el puerto 8081 y el frontend en Vite en el puerto 5173 con un proxy a 8081.

Comparación dev vs prod

Desarrollo

Backend con mvn spring-boot:run en puerto 8081. Vite dev server en 5173 con proxy de /baqueano/* al backend. BD baqueano directa por HikariCP. Swagger UI disponible.

Producción

Un único WAR de 56 MB en WildFly. SPA y API se sirven desde http://host/baqueano/. Datasource vía JNDI gestionado por WildFly. Logs en JSON. Swagger deshabilitado.

4. Capas del backend

Arquitectura clásica en capas con responsabilidades bien definidas. Las DTOs aíslan los controllers de las entidades JPA, y los mappers (MapStruct) son la única conexión entre ambos mundos.

Decisiones clave

  • Auditoría centralizada en Auditable (@MappedSuperclass) con campos fecha_creacion, fecha_modificacion, usuario_creacion, usuario_modificacion.
  • AuditorAware<String> lee del SecurityContextHolder; cae a "system" en runners/jobs.
  • DTOs como Java record inmutables, con @NotBlank/@Email/etc. en los componentes.
  • Servicios readOnly = true por clase, override en métodos de escritura.
  • SubmenuPerfil con clave compuesta @EmbeddedId + @MapsId.
  • Cache local (ConcurrentMapCacheManager) para parametros: @Cacheable en lectura, @CacheEvict(allEntries=true) en escritura.

5. Modelo de datos

Ocho tablas en MySQL 8 (InnoDB / utf8mb4_unicode_ci). Toda escritura va por Flyway: el schema está bajo baqueano-backend/src/main/resources/db/migration/ con migraciones V1__schema_inicial.sql, V2__seed.sql y V3__refresh_tokens.sql. spring.jpa.hibernate.ddl-auto = validate.

Menú dinámico: el árbol que recibe el frontend en /api/v1/menu/mio se construye con una query JPQL que une submenu + menu + submenu_perfil filtrando por el perfil_id del usuario logueado y puede_ver = true. Cambiar permisos en BD se refleja en el siguiente login (o refresh del menú) sin redeploy.

6. Flujo de autenticación

Características de seguridad

  • JWT HS256, claims sub/uid/perfilId/perfil/iat/exp, 30 min de vida.
  • Refresh token aleatorio de 32 bytes (Base64 url-safe). En BD se guarda solo el SHA-256.
  • BCrypt cost 10 para passwords. AdminPasswordRunner regenera el hash placeholder en el primer arranque.
  • Bloqueo por intentos: contador en usuarios.intentos_fallidos; máximo desde parametros.login.intentos.max.
  • Stateless en backend (SessionCreationPolicy.STATELESS), CSRF off, CORS solo para localhost:5173 en dev.
  • Errores uniformes: { timestamp, status, error, code, message, path, fieldErrors? }. 401 con código (CREDENCIALES_INVALIDAS, USUARIO_INACTIVO, USUARIO_BLOQUEADO).
  • Autorización fina: @PreAuthorize("@permisoEvaluator.puede(authentication, '/usuarios', 'CREAR')").

7. Entorno y despliegue

Comandos de despliegue

# 1) Instalar el driver MySQL como módulo de WildFly (una vez)
.\scripts\wildfly\setup-mysql-module.ps1

# 2) Levantar WildFly
& "$env:WILDFLY_HOME\bin\standalone.bat"

# 3) Crear datasource BaqueanoDS
& "$env:WILDFLY_HOME\bin\jboss-cli.bat" --connect `
  --file=scripts\wildfly\wildfly-datasource.cli

# 4) Setear perfil prod (editar standalone.conf.bat)
set "JAVA_OPTS=%JAVA_OPTS% -Dspring.profiles.active=prod"
set "JAVA_OPTS=%JAVA_OPTS% -DJWT_SECRET=ALGUN_SECRETO_REAL"

# 5) Hot deploy
Copy-Item baqueano-backend\target\baqueano.war `
  "$env:WILDFLY_HOME\standalone\deployments\"

Recorrido típico del ADMIN

  1. Login en /login con admin/admin123.
  2. Dashboard muestra totales (usuarios, contactos nuevos, perfiles) y últimos 5 contactos.
  3. Click en "Gestión de Usuarios" → ABM con paginación y botones Nuevo/Editar/Eliminar.
  4. Alta de un nuevo usuario con perfil GESTOR → validaciones del lado cliente + servidor.
  5. Verifica que el usuario aparece en la tabla → modifica algún campo → confirma.
  6. Baja lógica de un usuario (no se elimina la fila, queda activo=false).
  7. Logout → vuelve a /login; refresh token queda revocado en BD.

Recorrido del usuario CONSULTA

  1. Login → ve sidebar SIN Usuarios ni Parámetros.
  2. Puede ver Dashboard, Perfiles, Contactos en modo lectura.
  3. Si intenta acceder a /usuarios manualmente, la API responde 403; en UI los botones de acción están ocultos.

10. Manual de usuario — ABM Usuarios

Esta sección muestra, paso a paso, cómo administrar usuarios desde la UI. Disponible para perfiles con permiso VER /usuarios (en el seed, solo el perfil ADMIN).

Pantalla principal

Crear un usuario

  1. Hacer click en el botón "+ Nuevo" (esquina superior derecha). El botón solo está visible si tu perfil tiene permiso CREAR en la ruta /usuarios.
  2. Se abre un modal con los siguientes campos:
    CampoTipoReglas
    Usernametextoobligatorio, máx 50, único en el sistema
    Emailemailobligatorio, formato válido, máx 150, único
    Contraseñapasswordobligatorio, mínimo 8 caracteres, máx 100. Se hashea con BCrypt antes de guardar.
    Nombretextoobligatorio, máx 100
    Apellidotextoobligatorio, máx 100
    Perfilselectobligatorio. Select cargado desde /api/v1/perfiles.
    Activocheckboxdefault: marcado
  3. Si hay errores de validación, aparecen en rojo bajo cada campo y el submit se bloquea (no se llega a llamar al backend).
  4. Al hacer click en "Guardar", la SPA envía POST /api/v1/usuarios con el DTO. Posibles respuestas:
    • 201 Usuario creado. Toast verde "Usuario creado". El modal se cierra y la tabla recarga.
    • 409 OPERACION_INVALIDA Username o email duplicado. Toast rojo con el mensaje específico.
    • 400 VALIDATION_ERROR Algún campo no cumple las reglas del backend.
    • 403 FORBIDDEN Tu perfil no tiene permiso CREAR /usuarios.

Editar un usuario

  1. Click en el ícono ✎ de la fila a editar. Solo visible con permiso EDITAR.
  2. La SPA hace GET /api/v1/usuarios/{id} para traer el objeto completo (incluye perfilId, que no está en el list-item).
  3. Se abre el modal con los datos precargados. Importante: en modo edición, los campos Username y Contraseña NO se muestran (el username es inmutable; la contraseña tendrá su propio endpoint de cambio).
  4. Modificar los campos deseados y hacer click en "Guardar" → PUT /api/v1/usuarios/{id}.
  5. Toast verde "Usuario actualizado". Si el email cambia y choca con otro existente, se muestra el toast 409.

Dar de baja un usuario

El borrado es baja lógica: la fila no se elimina, solo se setea activo = false. Esto preserva la integridad referencial con tablas de auditoría y permite reactivación posterior.
  1. Click en el ícono 🗑 (papelera roja) de la fila. Solo visible con permiso ELIMINAR.
  2. Aparece un diálogo de confirmación: "¿Dar de baja a 'X'? La operación es reversible (baja lógica)."
  3. Click en "Dar de baja" → DELETE /api/v1/usuarios/{id}.
  4. Toast verde "Usuario dado de baja". La tabla recarga y la fila aparece con badge Inactivo.
  5. Para reactivar: editar el usuario, marcar el checkbox Activo, guardar.

Errores y troubleshooting comunes

SíntomaCausa probableSolución
No veo el botón "Nuevo" Tu perfil no tiene CREAR /usuarios en submenu_perfil Pedir al admin que actualice los permisos en BD o cambiar de perfil
Toast rojo "Email ya existe" El email ingresado ya está usado por otro usuario activo o inactivo Usar un email distinto, o reactivar el usuario existente
Toast rojo "Credenciales inválidas (CREDENCIALES_INVALIDAS)" El JWT expiró y el refresh también Volver a hacer login
Página recargada vuelve a /login localStorage limpio o sesión expirada Login normal; el SPA recordará la URL anterior
"Usuario bloqueado por intentos fallidos" Se superó parametros.login.intentos.max (default 5) Admin debe entrar a la base y resetear usuarios.intentos_fallidos = 0
